応用編 A9

自動化レシピ集

このチャプターのゴール

  • これまで学んだ機能を 組み合わせて 実用ワークフローを作る
  • すぐ真似できる 5 つのレシピを手元に持って帰る

レシピ 1: 「保存するだけで型チェック・整形・テスト」

Claude が編集を加えたら、自動で format → typecheck → 関連テスト実行までやらせるレシピ。 失敗があれば Claude に戻って自動で修正に動いてくれます。 

.claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit",
        "hooks": [
          {
            "type": "command",
            "command": "pnpm format && pnpm typecheck && pnpm test --run --changed"
          }
        ]
      }
    ]
  }
}

フック 1 個で完結。 失敗ログがそのまま Claude に渡るので、Claude 自身が対応に動きます。

レシピ 2: 「PR ドラフトをワンコマンドで」

ブランチで作業した内容をまとめて、PR タイトル・本文・テストプランを自動生成するワークフロー。 

.claude/commands/pr.md
---
description: PR の説明を生成し、gh pr create で送信する
allowed-tools: Bash
argument-hint: "[base-branch (default: main)]"
---

base ブランチ: $ARGUMENTS (空なら main)

以下の手順で進めてください。
1. 現在のブランチ名と差分を確認
   - !`git branch --show-current`
   - !`git diff origin/${ARGUMENTS:-main}...HEAD --stat`
2. 差分を読んで PR 本文を Markdown で起稿
   セクション: 概要 / 変更点 / テスト方法 / リスク
3. `gh pr create --base ${ARGUMENTS:-main} --title "..." --body "..."`
   を実行して PR を作る

呼び出すときは /pr または /pr develop のように。 カスタムコマンド + Git + GitHub CLI の合わせ技です。

レシピ 3: 「セキュリティガード — 機密情報を絶対に読ませない・送らせない」

.claude/settings.json
{
  "permissions": {
    "deny": [
      "Read(.env)",
      "Read(.env.*)",
      "Read(**/secrets/**)",
      "Read(**/*.pem)",
      "Read(**/*.key)",
      "Bash(cat .env:*)",
      "Bash(env)",
      "Bash(printenv:*)"
    ]
  },
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "node -e \"const i=JSON.parse(require('fs').readFileSync(0));const cmd=i.tool_input.command;if(/AKIA|sk-ant|ghp_|github_pat_/i.test(cmd)){console.error('blocked: secret-like pattern');process.exit(2)}\""
          }
        ]
      }
    ]
  }
}

ポイントは 2 重防御。permissions.deny で読ませず、 フック でコマンド内容に機密パターンが現れたら強制ブロック。

レシピ 4: 「コードレビュー専門エージェント+定期実行」

サブエージェント として code-reviewer を定義し、 カスタムコマンド から呼び出す。 コミット直前にチェックする習慣が作れます。 

.claude/agents/code-reviewer.md
---
name: code-reviewer
description: ステージ済みの差分を読んでレビューする。コミット前のチェックに使う。
tools: Read, Grep, Glob, Bash(git diff:*), Bash(git status)
---

シニアレビュアとして以下を確認:
- バグ・null 安全性・型不整合
- 命名の一貫性
- テストの抜け
- セキュリティ (SQLi, XSS, 機密漏れ)

出力は Critical / Suggestion / Nits の 3 セクション。
何も問題なければ「✅ 問題ありません」とだけ。
.claude/commands/review.md
---
description: ステージ済みの差分をレビュー
---

code-reviewer サブエージェントを使って、いまステージされている差分をレビューしてください。
レビュー後、修正すべき指摘があれば自分でも直してください (Critical のみ)。

レシピ 5: 「終了したら通知 + ログを残す」

長時間の作業を頼んでパソコンを離れたいときに便利。 Stop フックで通知 + 監査ログをファイルに残します。 

{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display notification \"Claude が応答完了\" sound name \"Pop\"'"
          },
          {
            "type": "command",
            "command": "echo \"$(date -u +%FT%TZ) session=$(jq -r .session_id) stop\" >> ~/.claude/audit.log"
          }
        ]
      }
    ]
  }
}

Windows なら powershell -Command "..."、Linux なら notify-send に置き換え。

組み合わせの考え方

自動化を組み立てるとき、目的別に道具を選ぶと迷いません。

やりたいこと適した道具
毎回同じ依頼をしたいカスタム スラッシュコマンド
専門領域を切り出したいサブエージェント
ある操作の前後に何かしたいフック
外部システムと連携したいMCP
特定領域の知識を渡したいスキル
常時のお願いをしたいCLAUDE.md
強制ルールを敷きたいsettings.json deny + フック

運用のコツ

  • 少しずつ追加する — フックや許可ルールを一度に大量に書くと、原因切り分けが大変
  • チームに共有する — .claude/ を Git にコミット (ローカル設定は除く)
  • レビュー対象にする — 設定ファイル・フック・サブエージェント定義の変更は PR レビューの対象に
  • Claude に診断してもらう — 「この settings.json と hooks にリスクがあれば指摘して」と聞くと効果的
自動化を増やしすぎると 「Claude が何をしているのか分からない」 状態に陥りがちです。 自動で動くものには「なぜそれが動いたか」がログから追えるようにしておきましょう。

このチャプターのまとめ

  • 5 つのレシピ: フォーマット&検証・PR 自動化・機密ガード・レビューエージェント・通知ログ
  • 道具を目的別に選び、少しずつ積み上げる
  • 設定の変更も Git 管理 / レビュー対象に

応用編のまとめ

ここまで読んだあなたは、Claude Code を 自分の働き方に合わせて作り変える ための 道具を一通り持っています。

  • プロジェクト運用と CLAUDE.md で土台を整え
  • カスタムコマンド・サブエージェント・スキルで「型」を作り
  • フックと許可ルールで安全網を張り
  • MCP で外の世界とつなぐ

最初はひとつだけ ── 自分が一番繰り返している作業に対して、レシピを 1 つ書いてみてください。 そこから自然と「これも自動化したい」が増えていきます。
