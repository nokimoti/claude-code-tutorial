応用編 A1

プロジェクト運用のコツ

このチャプターのゴール

  • 複数プロジェクトを行き来する流れをつかむ
  • 「会話を切る/続ける」の判断基準を持つ
  • プロジェクト固有の設定とユーザー設定を使い分ける

プロジェクト = フォルダ単位で考える

Claude Code は プロジェクトフォルダ 単位で 設定や履歴を管理します。フォルダの中に .claude/ という隠しフォルダが作られ、ここに

  • settings.json — チームと共有する許可ルール / フック
  • settings.local.json — 自分専用 (Git に入れない)
  • commands/ — カスタム スラッシュコマンド
  • agents/ — サブエージェント
  • skills/ — スキル

が入ります。フォルダごと別のパソコンに移しても、同じ環境を再現できる作りです。

3 階層の設定

Claude Code の設定は、3 階層で重ね合わせる仕組みになっています (上ほど優先)。

階層場所用途
個人ローカル.claude/settings.local.json自分専用、Git 除外
プロジェクト.claude/settings.jsonチームで共有、Git に入れる
ユーザー~/.claude/settings.jsonパソコン全体に効く好み

たとえば「npm test をいつも許可」はチーム共有 (プロジェクト)、 「日本語で答えて」は個人ユーザー設定、というふうに役割を分けます。

会話の境界線をどこに引くか

ベテラン使いになるほど こまめに会話を区切る ようになります。 一つの会話に詰め込みすぎないことが、応答精度と料金の両方にプラスに働きます。

区切るべきタイミング

  • 機能開発が一区切りついた
  • 調査が終わって実装に入る
  • バグ修正が終わってリファクタに移る
  • 異なるパッケージや別のディレクトリの作業に移る

続けるべきタイミング

  • 同じ機能の調整・微修正を続けているとき
  • テストの修正と本体の修正を行き来しているとき
  • 「さっきの差分」「いま書いた関数」のように直前の文脈を使うとき
会話を切る目安: 上のサイドバーで /context や使用率が 50% を超えてきたら /clear または /compact を検討。100% 近くなる前に手を打つのがコツです。

同じプロジェクトで「並行作業」する

バグ修正をしている最中に、別のブランチで実験もしたい ── そんなときは Git の worktree や別のクローンを使い、 Claude Desktop アプリを 2 つ起動して別フォルダを開く のが定番です。

アプリは同時に複数起動できる作りになっており、それぞれが別のセッションを持ちます。 履歴もごちゃ混ぜになりません。

新しいプロジェクトを Claude に渡すときの流れ

  1. アプリで「フォルダを開く」 → プロジェクトを選ぶ
  2. 入力欄に /init を打って CLAUDE.md を作ってもらう
  3. 提案された CLAUDE.md を読んで「ビルドコマンドが違う」「テストランナーが違う」など修正を依頼
  4. /permissions で頻繁に使うコマンド (git status など) を許可リストに追加
  5. 「このプロジェクトの全体像を 5 行で教えて」と聞いて、Claude の認識が正しいか確認

この最初の 5〜10 分の投資で、その後数時間〜数日の作業が劇的に楽になります。

機密情報の扱い

API キーやパスワードを含むファイルをアプリに見せない のが鉄則です。 .env や .env.local、認証ファイルは settings.json の deny ルールに入れて、 Claude が間違って読まないようにしておきます。 
{
  "permissions": {
    "deny": [
      "Read(.env)",
      "Read(.env.*)",
      "Read(secrets/**)",
      "Read(**/*.pem)"
    ]
  }
}

このチャプターのまとめ

  • 1 プロジェクト = 1 フォルダ。中の .claude/ に設定が集まる
  • 設定は「個人ローカル / プロジェクト / ユーザー」の 3 階層
  • 区切れる作業はこまめに /clear
  • 新規プロジェクトはまず /init から
  • 機密ファイルは deny ルールで Claude から守る
