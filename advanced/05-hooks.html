応用編 A5

フック実践

このチャプターのゴール

  • 主要な フック の発火タイミングを把握する
  • 「自動フォーマット」「禁止操作のブロック」「終了通知」を実装できる
  • フックの戻り値で Claude の挙動に介入できることを知る

フックは「Claude のイベントに割り込む」仕組み

フック を設定すると、Claude が ツールを使う前後 / メッセージ送信時 / セッション開始時 などのイベントに合わせて、 あなたが指定したコマンド が自動で走ります。 設定は settings.json に書きます。

主なフックタイプ

イベント名発火タイミング主な用途
PreToolUseツール実行 前禁止操作の事前ブロック、ログ記録
PostToolUseツール実行 後編集後のフォーマット、ビルドチェック
UserPromptSubmitユーザー送信時プロンプトに自動でコンテキストを足す、検閲
StopClaude が応答完了時完了通知、ログ記録
SubagentStopサブエージェント完了時結果ログ、後処理
SessionStartセッション開始時状態の表示、最新化チェック
SessionEndセッション終了時クリーンアップ、サマリ送信
Notification通知発生時 (待ち状態など)OS 通知の発火

最小例: 編集後に自動フォーマット

.claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit",
        "hooks": [
          { "type": "command", "command": "pnpm prettier --write ." }
        ]
      }
    ]
  }
}

matcher は対象とするツール名の正規表現。Edit / Write / MultiEdit が走ったあと、 pnpm prettier --write . が自動で実行されます。

matcher の書き方

  • "Edit" — 完全一致
  • "Edit|Write" — どちらかにマッチ
  • "Bash" — Bash ツール全部
  • "Bash:rm.*" — Bash の中で rm 系コマンドだけに反応 (一部のフックタイプで対応)
  • "" — すべてのツールにマッチ (空文字列)

フックが受け取る情報

フックコマンドには、JSON が標準入力 (stdin) で渡されます。 これを読むと、どのツールがどんな引数で呼ばれたかがわかります。 

{
  "session_id": "abc123",
  "tool_name": "Edit",
  "tool_input": {
    "file_path": "/path/to/file.ts",
    "old_string": "...",
    "new_string": "..."
  }
}

シェルスクリプトなら jq を使って取り出せます。 

FILE=$(jq -r '.tool_input.file_path')
case "$FILE" in
  *.ts|*.tsx) pnpm prettier --write "$FILE" ;;
  *.py)       black "$FILE" ;;
esac

フックの「戻り値」で挙動をコントロール

フックコマンドの 終了コード が、Claude の挙動に影響します (主に PreToolUse)。

終了コード意味
0OK。そのまま続行
2ブロック。Claude にエラーを通知して再考させる
その他非ブロッキングのエラー (ログだけ残る)

また stdout に JSON を返すと、より細かい制御ができます。 

{
  "decision": "block",
  "reason": "本番環境への push は禁止されています"
}

レシピ集

1. .env ファイルの読み取りを禁止

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Read",
        "hooks": [
          {
            "type": "command",
            "command": "node -e \"const i=JSON.parse(require('fs').readFileSync(0)); if(/\\\\.env/.test(i.tool_input.file_path)){console.error('blocked');process.exit(2)}\""
          }
        ]
      }
    ]
  }
}

2. 編集が走ったら型チェック

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit",
        "hooks": [
          { "type": "command", "command": "pnpm typecheck 2>&1 | tail -20" }
        ]
      }
    ]
  }
}

出力は Claude にコンテキストとして戻るので、エラーがあれば自動で修正に動いてくれます。

3. 応答完了時に通知

{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          { "type": "command", "command": "osascript -e 'display notification \"Claude が応答を終えました\"'" }
        ]
      }
    ]
  }
}

macOS の例。Windows なら powershell -c "..." で BurntToast や msg.exe を使う。

4. プロンプト送信前に Git の状態を自動添付

{
  "hooks": {
    "UserPromptSubmit": [
      {
        "matcher": "",
        "hooks": [
          { "type": "command", "command": "git status -s" }
        ]
      }
    ]
  }
}

毎回、現在の git ステータスがプロンプトの先頭に追加されます。 Claude が「いま何が変わっているか」を常に把握した状態で答えてくれます。

デバッグ方法

  • 入力欄で /hooks を打つと、設定済みのフック一覧が表示される
  • 標準出力 / 標準エラーの内容は Claude に渡される (確認しやすい)
  • テストはコマンド単体で動かす: echo '{"tool_input":{"file_path":"x.ts"}}' | (フックコマンド)
フックは ユーザー権限で勝手に動く 仕組みです。 .claude/settings.json はチームで共有されるので、悪意あるフックが混入しないか git diff で確認する習慣をつけましょう。 Claude Code は新規フックを検知すると確認ダイアログを出します。

このチャプターのまとめ

  • フックはイベントに割り込む仕組み (PreToolUse / PostToolUse / Stop など多数)
  • matcher でツール / コマンドを絞り込める
  • 標準入力で JSON、終了コード 2 でブロックできる
  • 自動フォーマット・型チェック・禁止操作・通知などに使える
