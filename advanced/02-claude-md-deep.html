応用編 A2

CLAUDE.md 詳細

このチャプターのゴール

  • CLAUDE.md の階層的読み込みのしくみを知る
  • 「効く」CLAUDE.md と「効かない」CLAUDE.md の違いを理解する
  • 分割・インポートで肥大化を防ぐ

CLAUDE.md は重ね合わせで読み込まれる

CLAUDE.md は単一のファイルではなく、 複数の階層から自動で読み込まれて連結 されます。

  1. ~/.claude/CLAUDE.md — ユーザー個人 (パソコン全体)
  2. ./CLAUDE.md — プロジェクトルート
  3. ./(サブフォルダ)/CLAUDE.md — サブフォルダ専用 (そこを触るときだけ)
  4. ./CLAUDE.local.md — 個人専用 (Git に入れない)

たとえば src/auth/login.ts を編集する際は、ルートの CLAUDE.md と src/auth/CLAUDE.md の両方が効きます。 モノレポや巨大プロジェクトで「この領域に固有のルール」を局所的に書けるのが利点です。

「効く」 CLAUDE.md の 3 原則

1. 短く、具体的に

Claude は CLAUDE.md を 毎回プロンプトに含めて 動きます。 長すぎると トークン を食うだけでなく、 重要な指示が埋もれて 無視されやすくなり ます。 理想は 200 行以内、最大でも 500 行を目安に。

2. 「やる/やらない」で書く

抽象的な目標より、具体的な動作指示のほうが効きます。

  • ❌ 「コードは美しく書いてください」
  • ✅ 「セミコロンは必ず付ける。インデントはスペース 2 つ。1 行 100 文字以内」

3. 失敗例を入れる

「やってはいけないこと」を 1〜2 行入れると効果が高いです。 

# 注意

- console.log を残したままコミットしない
- TypeScript で any を新規追加しない (既存の any はそのまま)
- 関数を全部 export default にしない (named export を使う)

パターン集: プロジェクト用 CLAUDE.md

Web アプリのバックエンド

# プロジェクト概要
社内向けタスク管理アプリの API サーバー。

# 技術スタック
- TypeScript / Node.js 20
- Express + Drizzle ORM
- PostgreSQL 15
- テスト: Vitest

# 開発コマンド
- 開発サーバ: `pnpm dev`
- テスト: `pnpm test`
- 型チェック: `pnpm typecheck`
- DB マイグレーション: `pnpm db:migrate`

# コーディングルール
- インポートはエイリアス (`@/...`) ではなく相対パス
- ルートハンドラは `src/routes/` に置く
- DB 操作はリポジトリ層 (`src/repos/`) 経由でのみ

# Claude へのお願い
- 大きい変更はまず計画モードで提案
- DB スキーマを変えるときは必ずマイグレーション付きで
- ルートを足したら同じファイルにテストも書く

個人ユーザー (~/.claude/CLAUDE.md)

# 言語と口調
- 日本語で答える
- 敬語ではなく丁寧語 (です・ます調)
- コードコメントは英語で OK

# デフォルトの好み
- パッケージマネージャは pnpm (なければ npm)
- バンドラは Vite を優先
- リンタは ESLint Flat config を優先

# やらないでほしいこと
- 大量のコメントを足さない
- ファイルの末尾に「Generated by AI」のような注釈を入れない

分割とインポート

CLAUDE.md には他のファイルを読み込むための @ 記法 があります。 

# プロジェクト概要
@docs/overview.md

# コーディングルール
@docs/coding-rules.md

# Claude へのお願い
- (短い指示はここに直接書く)

@パス と書くと、その内容が読み込まれた瞬間に展開されます。 長くなる説明は別ドキュメントに切り出して、CLAUDE.md は目次のような役割にできます。

動的に追記する: # プロンプト

入力欄で # から始めて何か書くと、そのまま CLAUDE.md に追記されます。 

# テストは必ず日本語のテストケース名で書いて

作業中に「あ、このルール覚えておいてほしい」と気づいた瞬間に書けば、 次回以降のセッションでも有効になります。

アンチパターン

  • サンプルコードを大量に貼る — Claude は具体的な参照より、ルールのほうが効きやすい。 サンプルは別ファイルにしてリンクで指す。
  • 「ベストプラクティスに従って」のような抽象指示 — 何のベストプラクティスかが不明瞭。 具体ルールに落とす。
  • 毎回トリビアルに書き直す — 一度入れたルールは Claude が引きずる。 変えるときは「以前のルールを上書き」と明示する。
  • 機密情報を書く — CLAUDE.md は Anthropic に送信されます。API キーや顧客データを書かない。
ドッグフーディング: Claude に「この CLAUDE.md を読んで、改善点を 3 つ提案して」と聞くと、 矛盾や曖昧さを指摘してもらえます。定期的に見直しの相手をしてもらいましょう。

このチャプターのまとめ

  • CLAUDE.md は階層から重ね合わせで読まれる
  • 短く・具体的に・「やる/やらない」で書く
  • 長くなる場合は @ファイル で分割
  • # プロンプトで動的に追記できる
  • 機密情報は絶対に書かない
