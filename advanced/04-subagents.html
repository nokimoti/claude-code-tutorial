応用編 A4

サブエージェントを使いこなす

このチャプターのゴール

  • サブエージェントが解決する 2 つの課題を理解する
  • 役割定義ファイルを書ける
  • 「自動呼び出し」と「明示呼び出し」を使い分ける

サブエージェントが解く 2 つの課題

サブエージェント は、メインの Claude が 仕事の一部を別の Claude に委譲する ための仕組みです。 これが解決するのは主に 2 つの課題です。

  1. コンテキスト汚染を避ける — 巨大なログを読ませる、ライブラリ全体を調査する、 などの作業はメインの会話を圧迫します。サブエージェントは独立した コンテキストウィンドウ を持つので、 最終結果だけがメインに戻ります。
  2. 役割を専門化する — 「コードレビュー専門」「テスト実行専門」のように 指示と権限をパッケージ化しておくと、毎回プロンプトを書き直す手間が減ります。

定義ファイルの場所

  • プロジェクト用: .claude/agents/(名前).md
  • ユーザー用: ~/.claude/agents/(名前).md

アプリの入力欄で /agents と打つと、新規作成・編集・削除を GUI で扱うこともできます。

定義ファイルの形

---
name: code-reviewer
description: コードの差分をレビューし、問題点を箇条書きで返す。実装の前後、PR 直前に使う。
tools: Read, Grep, Glob, Bash(git diff:*)
model: claude-sonnet-4-5
---

あなたはシニアエンジニアとしてコードレビューを行います。

# 観点
- 明らかなバグ・null 安全性・型の不整合
- 命名の一貫性
- テストカバレッジの抜け
- セキュリティ (SQL インジェクション、XSS、機密情報漏れ)

# 出力形式
以下の Markdown で返してください。

## Critical (要修正)
- ...

## Suggestion (提案)
- ...

## Nits (好み)
- ...

何も問題がなければ「✅ 問題ありません」とだけ返してください。

フロントマターの主要キー

キー意味
nameサブエージェントの識別名 (英小文字とハイフン推奨)
descriptionもっとも重要。Claude が「いつこのエージェントを呼ぶか」を判断する手がかり
toolsこのエージェントが使える ツール を限定 (省略時は親と同じ)
model使うモデルの上書き。軽い作業は Haiku、難しい設計は Opus、など
description は 具体的なトリガー条件を含める のがコツです。 「実装の前後、PR 直前に使う」のようにいつ呼ばれるかを書くと、 Claude が自動で適切なタイミングで呼び出してくれます。

2 つの呼び出し方

1. 自動呼び出し

会話の中で「この変更をレビューして」のような依頼があると、 Claude が description を見て、合致するサブエージェントを自動で呼びます。 呼ばれたかどうかは画面の「● Task(code-reviewer)」のような表示でわかります。

2. 明示呼び出し

「code-reviewer サブエージェントを使ってレビューして」のように 名前を明示しても呼び出せます。 自動で呼ばれてほしくないが、必要なときだけ確実に使いたい場合に便利です。

定番のサブエージェント例

code-reviewer

差分を読んで、改善点を箇条書きで返す。実装後、コミット前のチェックに最適。

test-runner

テストを実行し、失敗だけを抜き出して原因を要約。長いログをメインに持ち込まずに済む。

debugger

エラーログとコードを読んで、原因を仮説立てる。実装は行わず、診断のみ。

doc-writer

関数やモジュールの説明文を生成。コメントを書きすぎないように制約をかけられる。

researcher

外部 API ドキュメントなどを読み込んで、要点だけ持ち帰る。WebFetch を多用する作業向き。

権限を絞ることで安全に

サブエージェントの tools を限定すると、想定外の操作を防げます。 たとえば code-reviewer に Edit や Write を渡さないことで、 「レビュー中に勝手にコードを書き換えられる」事故を防げます。 

---
name: code-reviewer
tools: Read, Grep, Glob, Bash(git diff:*)
---

Bash(git diff:*) のように、Bash ツールでも特定コマンドだけを許可できます。

呼び出しの結果はどう戻る?

サブエージェントが返した 最終メッセージのみ がメインの会話に戻ります。 途中の調査ログや試行錯誤はメインのコンテキストに入りません。これがコンテキスト節約の核心です。

サブエージェントは ステートレス です。同じ会話で 2 回呼んでも、 前回のことは覚えていません。引き継ぎたい情報は呼び出し側のメッセージに含めます。

このチャプターのまとめ

  • サブエージェントはコンテキスト分離と役割分担のための仕組み
  • 定義は .claude/agents/*.md、フロントマターと本文で書く
  • description の書き方が自動呼び出しの精度を決める
  • tools を絞ると安全性が上がる
