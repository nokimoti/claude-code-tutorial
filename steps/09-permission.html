STEP 09

許可の仕組み

このステップのゴール

  • 4 つの 許可モード の違いを把握する
  • 自分の用途に合った安全レベルを選べるようになる

許可は「まとめて」設定もできる

STEP 07・08 で、ファイル編集やコマンド実行のたびに「Yes / No」を聞かれることを見ました。 でも、毎回押すのが面倒なときもあります。 Claude Code には、4 段階の 許可モード が用意されています。

4 つのモード

モード動き向いている用途
Default (標準) 編集・コマンド実行のたびに確認 慣れないうち、本番環境
Plan (計画) 調査・読み取りのみ。一切書き換えない 方針を相談したいとき (STEP 12 で詳しく)
Accept Edits ファイル編集は自動許可、コマンド実行は確認 大量に編集が続くと分かっているとき
Bypass (危険) すべて確認なし 使い捨ての環境、隔離されたコンテナ内

切り替え方

入力欄で Shift + Tab を押すと、4 つのモードを順番に切り替えられます。 画面の下にいま選んでいるモードが表示されているはずです。 

⏵⏵ accept edits on (shift+tab to cycle)

細かい許可ルールは settings.json で

「npm install はいつも許可」「rm は絶対に許可しない」のように、 コマンド単位で細かい許可ルールを書きたい場合は、 settings.json という設定ファイルを使います。

設定の優先順位はこの順:

  1. プロジェクトの .claude/settings.local.json (自分専用)
  2. プロジェクトの .claude/settings.json (チーム共有)
  3. ユーザーの ~/.claude/settings.json (パソコン全体)

たとえばこんなふうに書きます。 

{
  "permissions": {
    "allow": [
      "Bash(npm test:*)",
      "Bash(git status)",
      "Read(*)"
    ],
    "deny": [
      "Bash(rm -rf:*)"
    ]
  }
}

「Bypass」モードの注意

Bypass モード (--dangerously-skip-permissions) は強力ですが、 Claude が予期しないコマンドを実行しても止められません。 本番環境やメインの作業フォルダでは絶対に使わないでください。 使うなら Docker コンテナや使い捨ての VM の中だけに限定しましょう。

慣れるまでの推奨設定

  1. 最初の数日は Default モード のままで使う
  2. 毎回 1.Yes を押して、何が起きているか観察する
  3. 慣れてきたら Accept Edits に切り替えてみる
  4. 頻繁に許可するコマンド (git status など) は settings.json に登録する
入力欄で /permissions と打つと、いま許可されている操作と禁止されている操作の一覧が表示されます。

このステップのまとめ

  • 許可モードは Default / Plan / Accept Edits / Bypass の 4 段階
  • Shift + Tab で切り替えられる
  • 細かいルールは settings.json で書ける
  • Bypass は必要なときだけ、隔離された環境でだけ使う
